1. Painpoint

Painpoint trung tâm:

Nhà nghiên cứu mất hàng tuần để tổng quan tài liệu cho một chủ đề — đọc hàng trăm bài báo thủ công, và vẫn có nguy cơ bỏ sót công trình quan trọng hoặc không nhận ra khoảng trống nghiên cứu thực sự. Bất kỳ tool tự động hoá nào cũng phải đồng thời giải quyết: không bịa nguồn (hallucination), đủ rộng (coverage đa góc nhìn), và đủ nhanh (không bắt user chờ 10 phút cho 1 query).

#Painpoint cụ thểHệ quả nếu không xử lýNguồn/khảo sát chứng minh
1 Tool search thẳng mọi input, không phân biệt chat thông thường / câu hỏi mơ hồ / yêu cầu research thật Tốn API call + thời gian cho câu chào hỏi hoặc query quá ngắn ("RAG") không đủ tín hiệu để search hiệu quả; user không biết tool "đang nghĩ gì" trước khi search Đối chứng hành vi: Perplexity search ngay không hỏi lại; Elicit có dialog nhưng không sinh research plan có cấu trúc
2 Single-query, single-source search — 1 câu query trên 1 database duy nhất Corpus hẹp, thiếu đa dạng góc nhìn (foundational / optimization / evaluation / application / recent), bỏ sót paper nằm ở database khác hoặc tiếp cận từ hướng nghiên cứu khác LSE Study (arXiv:2603.20235, 2026): chỉ 20% overlap giữa bài AI chọn và bài chuyên gia con người chọn khi dùng single-query search
3 LLM-based research agent bịa citation (hallucinate DOI/paperId, hoặc suy diễn quá xa so với nguồn thật — citation drift) Bài viết ra trông hợp lý nhưng trích dẫn sai/không tồn tại — mất uy tín học thuật nếu lọt qua review Reference Hallucination in Deep Research Agents (arXiv:2604.03173): benchmark 13 LLM × 40 domain, tỷ lệ hallucinate citation từ 14% đến 95% tuỳ model/domain
4 Sinh nội dung + verify claim chạy tuần tự dù về bản chất là các task độc lập có thể chạy đồng thời 8 theme viết tuần tự (~80s) + 200 claim verify tuần tự (~400s) — tốn 5-10 phút không cần thiết cho 1 session, làm xấu trải nghiệm chờ So sánh AutoSurvey (1 LLM sequential, không có cơ chế parallel agent) ở cùng quy mô tác vụ

Non-goal — Research Gap Identification

Pipeline này không tự động phân loại/định nghĩa "khoảng trống nghiên cứu" theo cấu trúc (structured gap analysis, so sánh đối chiếu giữa các nghiên cứu cũ một cách hệ thống) — việc này quá chung chung nếu làm vội và dễ sai. Thay vào đó, hệ thống gợi ý gap bằng trực quan thông qua Knowledge Graph (Step ⑨bis, xem knowledge-graph.html) — claim layer với cụm contradicts dày quanh 1 theme là tín hiệu trực quan cho tension nghiên cứu, không phải kết luận tự động.

Công nghệ/APIopenai/gpt-oss-120b qua NVIDIA NIM (https://integrate.api.nvidia.com/v1), langchain-openai ChatOpenAI, temperature=0, stream=true cho SSE token streaming

Rules:

def reconstruct_abstract(inv: dict) -> str:           # OpenAlex không trả abstract text trực tiếp
    words = [""] * (max(max(p) for p in inv.values()) + 1)
    for word, positions in inv.items():
        for pos in positions: words[pos] = word
    return " ".join(words)

Step ①bis — Cross-source Deduplication

Công nghệ/APIrapidfuzz (fuzzy string matching)
Thuật toánPriority key tuần tự: DOI → arXiv ID → Semantic Scholar paperId → title fuzzy (last resort, threshold ≥90%); giữ bản ghi đầu tiên gặp trong danh sách hợp nhất (S2 thường có metadata tốt nhất nên đứng trước trong thứ tự merge)
Vấn đề giải quyếtMerge nhiều nguồn (S2+OpenAlex+arXiv) × nhiều sub-query chắc chắn sinh trùng lặp — không dedup sẽ làm sai lệch outline/MMR diversity và lãng phí ngân sách snowball/embed cho paper trùng
def dedup_cross_source(all_papers: list[dict]) -> list[dict]:
    seen = {}
    for paper in all_papers:
        if doi := paper.get("doi") or paper.get("externalIds", {}).get("DOI"):
            key = f"doi:{doi.lower()}"
        elif arxiv := paper.get("externalIds", {}).get("ArXiv"):
            key = f"arxiv:{arxiv}"
        elif pid := paper.get("paperId"):
            key = f"s2:{pid}"
        else:
            key = f"title:{normalize(paper['title'])}"
        if key not in seen:
            seen[key] = paper
    return list(seen.values())

Kết quả: ~500 bài raw (5 sub-query × 2 nguồn) → ~400-450 unique bài sau dedup; ở quy mô lớn hơn: 400-600 bài raw → 350-500 unique.

Step ④ — MMR Outline + User Approval

Công nghệ/APIMMR tự viết (mmr.py), openai/gpt-oss-120b temperature=0.7, LangGraph interrupt()
Thuật toánMMR-20 (Maximal Marginal Relevance, λ=0.5, fetch_k=150) chọn 20 paper đại diện đa dạng từ toàn corpus → LLM sinh 5-8 theme (mỗi theme có name+description) → interrupt({"themes": [...]}) tạm dừng cho user edit/approve → resume tiếp tục với theme đã chốt
Vấn đề giải quyếtOutline cần vừa liên quan (relevance) vừa đa dạng (không trùng ý) — top-k thuần similarity thường ra outline trùng lặp; user cần align kỳ vọng trước khi tốn thời gian viết nội dung thật
So sánh nhanhElicit: có dialog nhưng không sinh research plan/outline có cấu trúc; Gemini Deep Research: pattern tương tự (generate plan → user confirm) — xác nhận đây là pattern đúng cho deep-research tool
async def outline_gen_node(state):
    mmr_papers = mmr.mmr_select(query_vec=encode_query(state["query"]),
                                 fetch_k=MMR_PREFETCH_OUTLINE, k=20)
    outline_llm = llm.with_config({"temperature": OUTLINE_TEMPERATURE})  # 0.7
    draft = await outline_llm.ainvoke(OUTLINE_PROMPT + abstracts(mmr_papers))
    approved = interrupt({"type": "outline_approval", "outline": draft})
    return {"draft_outline": draft, "approved_outline": approved}

Step ⑤ — Per-theme Hybrid Search

Step ⑧ — Parallel Verifier Agents

Công nghệ/APIopenai/gpt-oss-120b temperature=0, S2 /snippet/search, ar5iv.labs.arxiv.org + BeautifulSoup, asyncio.gather theo batch
Thuật toán3-tier waterfall: Tier A — S2 snippet search trả đoạn văn bản gốc khớp claim (coverage ~30% paper); rỗng → Tier B — lấy full text HTML từ ar5iv (~70% paper arXiv có sẵn, parse BeautifulSoup, strip nav/header/footer, giữ 10k ký tự đầu, timeout 15s); lỗi/không có arXiv ID → Tier C — chỉ dùng abstract, conservative tuyệt đối (không bao giờ trả "Supported" nếu chỉ có abstract, vì abstract quá ngắn để chắc chắn). 5. Phương pháp test, đánh giá so sánh

  • E2E test: query thật "RAG optimization techniques" → chạy full flow Step 0→⑩, kiểm tra mọi field state được populate đúng
  • Test interrupt/resume: outline approval + claim review — tắt server giữa chừng, restart, resume đúng thread_id qua checkpointer
  • Đo wall-clock: Step ⑥ (Writer) + Step ⑧ (Verifier) mục tiêu < 2 phút tổng
  • Đo corpus size: mục tiêu 400-600 bài sau dedup (Step ①bis), 600-900 bài sau snowball (Step ②bis)
  • Đo hallucination rate: mục tiêu ~0% bịa paperId (RAG grounding)
  • Đo citation drift: mục tiêu giảm 65-70% sau 3-tier verification (Step ⑧)
  • Test SSE: stream không block, token hiện đúng thứ tự, step_start/step_done/llm_token khớp schema
  • Test checkpointer: đảm bảo Postgres/SQLite checkpointer giữ đúng state qua nhiều lần resume liên tiếp
