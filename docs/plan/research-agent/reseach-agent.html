1. Painpoint

Painpoint trung tâm:

Nhà nghiên cứu mất hàng tuần để tổng quan tài liệu cho một chủ đề — đọc hàng trăm bài báo thủ công, và vẫn có nguy cơ bỏ sót công trình quan trọng hoặc không nhận ra khoảng trống nghiên cứu thực sự. Bất kỳ tool tự động hoá nào cũng phải đồng thời giải quyết: không bịa nguồn (hallucination), đủ rộng (coverage đa góc nhìn), và đủ nhanh (không bắt user chờ 10 phút cho 1 query).

Non-goal — Research Gap Identification

Pipeline này không tự động phân loại/định nghĩa "khoảng trống nghiên cứu" theo cấu trúc — việc này quá chung chung nếu làm vội và dễ sai. Thay vào đó, hệ thống gợi ý gap bằng trực quan thông qua Knowledge Graph (Step ⑥, xem knowledge-graph.html) — claim layer với cụm contradicts dày quanh 1 theme là tín hiệu trực quan cho tension nghiên cứu, không phải kết luận tự động.

Step 0 — Query Intent Router + Research Plan

Kiến trúc (v2.0.1): 1 LLM call duy nhất với function calling thay cho 2 LLM calls tuần tự + regex parse. Node reply_generator_node bị loại bỏ hoàn toàn — reply/thinking/plan đều nằm trong arguments của function được gọi. Effort tỷ lệ với độ phức tạp input: greeting instant, conceptual brief, research full planning.
Công nghệ/APIopenai/gpt-oss-120b qua NVIDIA NIM, temperature=0, tool_choice="required"
Thuật toánFunction calling as intent routing — model chọn function nào để gọi chính là quyết định intent. 4 functions: send_greeting (greeting/off-topic), answer_question (câu hỏi khái niệm), ask_clarification (topic mơ hồ — hỏi 1 câu cụ thể), create_research_plan (topic rõ — sinh plan). Field thinking đặt đầu tiên — stream ra trước, user thấy model đang "suy nghĩ".
Anchored sub-queries (core_terms): trước khi sinh query, model trích core_terms = {required (+synonyms), context} — required là các khái niệm phân biệt của topic (VD "diffusion language model", "speculative decoding / drafter") kèm nhóm đồng nghĩa. Mỗi sub-query bắt buộc chứa ≥1 term required — không sinh query chung chung ("edge deployment", "benchmarking generative AI") làm tản chủ đề. refined_query = dạng canonical chặt; core_terms được truyền xuống Step ①bis để lọc độ liên quan

Step ① — Search + Dedup

Công nghệ/APISemantic Scholar Graph API (220M paper, CS/AI), OpenAlex REST (250M paper, humanities/social), PubMed (35M, nếu domain medical); httpx async, asyncio.gather, tenacity retry; rapidfuzz fuzzy matching cho dedup
Thuật toánSearch: N sub-queries × M sources đồng thời qua asyncio.gather. Per-source ceiling: quota = max_papers_total // len(sources) — tránh 1 nguồn áp đảo corpus, giữ theo relevance order từ API (không sort lại theo citationCount).
Dedup: Priority key: DOI → ArXiv ID → S2 paperId → title fuzzy (≥90%). Khi tìm thấy duplicate: merge fields còn thiếu vào bản giữ (scalar nếu None, externalIds merge cả 2 dict). Sau dedup: filter bỏ papers có abstract=None AND citationCount=None
Vấn đề giải quyếtSingle-source bỏ sót paper ở database khác; citationCount sort loại hết PubMed papers (citationCount=None); mỗi source mạnh ở fields khác nhau — không merge thì mất metadata

# Dedup + merge
def dedup(all_papers):
    seen = {}
    for paper in all_papers:
        key = get_key(paper)   # DOI → ArXiv → S2 id → title fuzzy
        if key not in seen:
            seen[key] = paper
        else:
            kept = seen[key]
            for field in ["abstract", "citationCount", "year", "url"]:
                if kept.get(field) is None: kept[field] = paper.get(field)
            kept["externalIds"] = {**paper.get("externalIds",{}), **kept.get("externalIds",{})}
    return [p for p in seen.values()
            if not (p.get("abstract") is None and p.get("citationCount") is None)]

Step ①bis — Relevance Filter (Hybrid RRF)

Bước lọc độ liên quan giữa query ↔ paper, chèn giữa Dedup và Embed. Giải bài toán: search đa nguồn + phân rã sub-query dễ kéo về paper lạc chủ đề (VD topic "diffusion drafter" lại lẫn paper về methyl hóa DNA / nhãn khoa / fraud). Chỉ giữ paper thật sự liên quan trước khi cluster + viết, nên corpus vào clustering là corpus đúng chủ đề.
Công nghệ/APIrank-bm25 (lexical), NVIDIA NIM nv-embed-v1 (semantic embedding của văn bản tự do — SPECTER không embed được query nên semantic side dùng NIM); Reciprocal Rank Fusion (RRF)
Thuật toán Hybrid search (2 tín hiệu chạy song song, cùng nhìn toàn corpus):
• BM25 xếp hạng toàn bộ paper (title+abstract) theo core_terms — bắt khớp thuật ngữ chính xác.
• Semantic: embed refined_query + mỗi paper bằng NIM → cosine — bắt paper diễn giải khác từ ("non-autoregressive parallel decoding" ≈ "diffusion drafter").
• RRF fuse: score(d) = Σ 1/(k + rank_i(d)) (k≈60) trộn 2 bảng xếp hạng → không cần ngưỡng cosine tuyệt đối (brittle). Giữ top-300; paper lạc (bét ở cả 2 bảng) tự rớt.
Sàn: nếu < relevance_min=20 paper qua lọc → gắn cờ low_relevance (nới lọc + Step ⑦ chèn caveat thành thật thay vì cố viết cho đầy)
Vấn đề giải quyếtCorpus lạc chủ đề → cluster trung thành gom rác thành theme gọn gàng nhưng sai đề → writer "bịa" liên hệ với topic. Lọc từ gốc chặn cả chuỗi. Hybrid (không phải cascade 2 tầng) để lexical + semantic bù nhau, không tín hiệu nào chặn trên recall của cái kia
Backing researchRRF (Cormack et al. 2009); Hybrid dense+sparse retrieval — BM25 bù chỗ mù ngữ nghĩa của embedding và ngược lại
GuardrailRELEVANCE_TOP_K=300 (trần cả corpus vào cluster), RELEVANCE_MIN=20 (sàn → low_relevance flag)
def relevance_filter(papers, refined_query, core_terms):
    docs = [f"{p.title} {p.abstract or ''}" for p in papers]
    bm25_rank    = rank_bm25(docs, query=core_terms)          # lexical, toàn corpus
    sem_rank     = cosine_rank(nim_embed(refined_query),
                               nim_embed_batch(docs))         # semantic, toàn corpus
    fused = rrf_fuse(bm25_rank, sem_rank, k=60)               # trộn thứ hạng
    kept  = fused[:RELEVANCE_TOP_K]                            # top-300
    low   = len(kept) < RELEVANCE_MIN                         # sàn 20 → cờ
    return kept, low

Lưu ý divergence: Step ①bis là extension so với bản v2.0 gốc (v2.0 không có relevance gate). Nó tái dùng ý tưởng hybrid BM25+semantic+RRF của hybrid_search cũ (đã bỏ ở Step ③) nhưng cho mục đích khác: lọc query↔paper relevance, không phải hybrid per-theme.

Step ③ — Clustering + User Approval (AutoSurvey)

Thay thế MMR-20 Outline + Per-theme Hybrid Search bằng clustering 4-pha theo phương pháp AutoSurvey (NeurIPS 2024, arXiv:2406.10252). Không cần encode query, không cần hybrid search per-theme.
Công nghệ/APIscikit-learn KMeans (k-means++ init), silhouette score, cosine similarity; openai/gpt-oss-120b temperature=0.7 để đặt tên cluster; LangGraph interrupt()
Thuật toán Pha 1 — Tìm k tối ưu: Thử k=3→10, tính silhouette score mỗi k, chọn k có score cao nhất.
Pha 2 — K-means clustering: Cluster paper vectors với k tối ưu (k-means++). Cluster <3 papers → discard.
Pha 3 — Rank + đặt tên: Sort papers theo cosine sim với centroid. Cap 15 papers/cluster (15 × ~150 token = ~2,250 token, tránh context rot). LLM sinh title + description.
Pha 4 — User approve: interrupt() hiện themes + papers → user đổi tên, merge, hoặc split (re-cluster k=2 tự động)
Vấn đề giải quyếtMMR không phát hiện cấu trúc thật của corpus. Hybrid search cần encode query (không có sau khi bỏ embed_text fallback). Clustering tìm theme từ chính data
Backing researchAutoSurvey (NeurIPS 2024, arXiv:2406.10252); Context rot (Chroma 2025) — LLM degradation với context dài; Silhouette score (Rousseeuw 1987)
def find_optimal_k(vectors, k_range=range(3, 11)):
    best_k, best_score = 3, -1
    for k in k_range:
        km = KMeans(n_clusters=k, init="k-means++", n_init=10, random_state=42)
        score = silhouette_score(vectors, km.fit_predict(vectors), metric="cosine")
        if score > best_score: best_k, best_score = k, score
    return best_k

Step ⑤ — Extract + Verify Claims

Công nghệ/APIExtract: regex fast-path + LLM fallback (temperature=0). Verify: openai/gpt-oss-120b temperature=0, S2 /snippet/search (qua s2_rate_limiter 1 req/s), ar5iv.labs.arxiv.org + BeautifulSoup, asyncio.gather (Semaphore 4 LLM calls)
Thuật toán Extract: Regex \[\[([A-Za-z0-9_]+)\]\] parse inline citations → sentence-level claims. Sau đó enrich mỗi claim với arxiv_id từ paper.externalIds["ArXiv"] (tra state["papers"]) để Verifier Case B có thể fetch full text.
Verify — 3-tier waterfall per claim:
Case A (~30%): S2 /snippet/search — calls qua s2_rate_limiter.acquire() (4 concurrent claims song song → 4 S2 req cùng lúc → vượt 1 req/s nếu không rate-limit).
Case B (~80% CS/AI): claim.arxiv_id có giá trị → fetch full text ar5iv, giữ 10k ký tự. Papers từ S2 vẫn có ArXiv ID dù đã bỏ arXiv làm search source.
Case C (fallback): Abstract conservative — không bao giờ trả "Supported". Claims Contrasting ưu tiên xử lý trước
Vấn đề giải quyếtCitation format [[PAPER_ID]] không thể biến thể → regex không miss. arxiv_id enrich ở Extract để Verify Case B hoạt động — nếu thiếu thì toàn bộ rớt xuống Case C (không verify được "Supported")
Backing researchLLM Cascade (Dohan et al. 2022); RouteLLM (Ong et al. 2024, arXiv:2406.18665)

Step ⑥ — Route + Human Review + KG Build

Công nghệ/APIRule-based routing (không gọi LLM), LangGraph interrupt(), networkx MultiDiGraph, python-slugify
Thuật toán Routing tự động: supported + partial → included; unsupported → removed.
Human review: Claims uncertain OR low_confidence (abstract-only) → interrupt() hiện full list kèm text, theme, paper.url. Resume value: {"approved": [...ids], "removed": [...ids]} → node merge vào đúng bucket. Claims Contrasting đứng đầu danh sách.
KG Build: Lắp ráp dữ liệu đã có thành multi-layer graph (Topic→Theme→Paper→Claim). 4 edge types: covers, belongs_to, evidenced_by, supports/contradicts/mentions. Không có cites edges (snowball đã bỏ)
Vấn đề giải quyếtClaims uncertain không nên tự động vào bài (rủi ro) cũng không tự động loại (có thể đúng) — cần user quyết định từng claim với đủ thông tin (link paper). Literature review văn xuôi tuyến tính khó nắm "ai đồng ý ai, ai phản bác ai" — xem knowledge-graph.html
async def route_claims_node(state):
    verified = state.get("verified_claims", [])
    included = [c for c in verified if c.status in ("supported","partial") and not c.low_confidence]
    removed  = [c for c in verified if c.status == "unsupported"]
    for_review = [c for c in verified if c.status == "uncertain" or c.low_confidence]
    for_review.sort(key=lambda c: c.intent != "Contrasting")  # Contrasting first

    paper_lookup = {p.paper_id: p for p in state["papers"]}
    resume = interrupt({"type": "claim_review", "claims": [
        {"id": c.id, "text": c.text, "theme": c.theme,
         "paper_url": paper_lookup.get(c.paper_id, {}).url}
        for c in for_review   # full list, không truncate
    ]})

    approved_ids = set(resume.get("approved", []))
    removed_ids  = set(resume.get("removed", []))
    included += [c for c in for_review if c.id in approved_ids]
    removed  += [c for c in for_review if c.id in removed_ids]
    return {"included_claims": included, "removed_claims": removed}

5. Phương pháp test, đánh giá so sánh

  • E2E test: query thật "RAG optimization techniques" → chạy full flow Step 0→⑦, kiểm tra mọi field state được populate đúng
  • Test interrupt/resume: cluster approval + claim review — tắt server giữa chừng, restart, resume đúng thread_id qua checkpointer
  • Đo wall-clock: Step ④ (Writer) + Step ⑤ (Verifier) mục tiêu < 2 phút tổng
  • Đo corpus size: mục tiêu 400-500 bài sau dedup (Step ①)
  • Đo hallucination rate: mục tiêu ~0% bịa paperId (RAG grounding)
  • Đo citation drift: mục tiêu giảm 65-70% sau 3-tier verification (Step ⑤)
  • Test SSE: stream không block, token hiện đúng thứ tự, step_start/step_done/llm_token khớp schema
  • Test checkpointer: giữ đúng state qua nhiều lần resume liên tiếp
  • Test claim review interrupt: resume value {"approved":[...], "removed":[...]} được xử lý đúng, bucket không overlap
