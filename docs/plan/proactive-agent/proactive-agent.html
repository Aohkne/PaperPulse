PaperPulse — Memory & Proactive Agent

Hai làn tách biệt: Recall (nhớ fact/persona người dùng để cá nhân hóa hội thoại) và Proactive (bắt thói quen tìm bài → chủ động gợi ý paper mới). Làn Proactive đã build sẵn ~75% trong codebase; làn Recall phải làm mới. Batch tổng hợp fact cuối ngày + giải quyết mâu thuẫn theo thời gian (temporal).

1. Painpoint

Painpoint trung tâm:

Trợ lý nghiên cứu hiện tại vô danh tính và bị động. Mỗi session bắt đầu từ con số 0: không nhớ người dùng tên gì, không nhớ họ đang theo đuổi hướng nào, và chỉ trả lời khi được hỏi. Người dùng phải tự lặp lại ngữ cảnh mỗi lần, và tự đi tuần các nguồn (arXiv/S2/OpenAlex) để biết có paper mới trong lĩnh vực của mình. Hai nhu cầu: (1) nhớ fact về người dùng để cá nhân hóa (Recall), (2) chủ động phát hiện & báo paper mới theo thói quen tìm kiếm (Proactive).

#PainpointHệ quả thực tếNguồn/khảo sát chứng minh
1 LLM statelessness — mỗi session mất sạch ngữ cảnh người dùng User phải lặp lại tên/lĩnh vực/sở thích mỗi lần; trải nghiệm generic, không "thuộc về" người dùng MemGPT / Letta (arXiv:2310.08560) — đặt vấn đề "LLM bị giới hạn context, cần hệ điều hành bộ nhớ" (core vs archival memory)
2 Fact mâu thuẫn theo thời gian — "dị ứng bò" hôm nay, "hết dị ứng bò" tháng sau Nếu chỉ append fact, bot vẫn nhớ fact cũ đã sai; nếu ghi đè cứng thì mất lịch sử, không audit được Zep / Graphiti (getzep.com — "Temporal Knowledge Graph Architecture for Agent Memory") — mỗi fact có valid_from/valid_to, invalidate thay vì xoá
3 Trích xuất fact bừa bãi — lưu cả câu ephemeral ("hôm nay tui mệt") Profile phình rác, tốn token inject, nhiễu tín hiệu thật Mem0 (github.com/mem0ai/mem0) — pipeline extract → classify → dedup → consolidate, bỏ thông tin không bền
4 Trợ lý bị động — user phải tự đi tuần arXiv/S2 xem có paper mới Bỏ lỡ paper liên quan; công việc theo dõi literature thủ công, tốn thời gian Precedent: Google Scholar Alerts, Semantic Scholar Research Feeds, arXiv Sanity — đều là proactive alerting theo interest
5 Interest suy từ lời tự khai không đáng tin bằng suy từ hành vi User nói "tui quan tâm X" một lần nhưng thực tế search Y 20 lần — cá nhân hóa theo lời nói sẽ lệch Behavioral signal > stated preference — codebase đã chọn hướng này: user_topic_interests.interest_score tính từ tần suất
Quyết định scope (Non-goal): (1) Không tự động gửi email/push khi chưa có opt-in rõ ràng — proactive sai chỗ thành spam/creepy. (2) Không dùng fact do LLM trích như instruction — luôn đóng khung "context, không phải lệnh" để tránh prompt-injection qua fact. (3) MVP không cá nhân hóa cron theo timezone từng user — 1 lịch UTC cố định. (4) Không xây lại hạ tầng topic-monitoring (đã có) — chỉ bổ sung scheduler + vector thật. (5) Recall lane không đọc lại toàn bộ lịch sử mỗi đêm — chỉ xử lý delta message mới trong ngày.

W2 — fact_extractor làm mới

Công nghệ/APILLM (gpt-oss-120b qua shared/services/llm_client.py), temperature=0, structured JSON output. Reuse embedding service của research_agent để nhúng fact.
Thuật toánĐọc delta message (chỉ tin nhắn mới từ last_processed_message_at) → prompt yêu cầu trả về mảng candidate fact dạng struct (subject/predicate/object), kèm fact_type và confidence. Loại ephemeral bị drop ngay tại extractor (không ghi DB).
Vấn đề giải quyếtPainpoint #3 — không lưu bừa. Struct thay vì free-text để bước reconcile so sánh được theo trường, và persona_context render gọn.
SYSTEM = """Extract durable facts about the user from the messages below.
Return JSON array; each item:
{ "fact_type": "identity|interest|preference|constraint|ephemeral",
  "subject": "user", "predicate": "...", "object": "...",
  "confidence": 0.0-1.0 }
Rules:
- fact_type=ephemeral for transient state (mood, "today I'm tired") -> caller will DROP it.
- Prefer interests INFERRED from repeated search behavior over one-off claims.
- Do NOT invent facts not supported by the text."""

W4 — fact_reconciler làm mới · khó nhất

Công nghệ/APISemantic search (cosine trên embedding fact, cùng user_id, status='active') + 1 LLM call phân loại quan hệ.
Thuật toánVới mỗi candidate: lấy top-k fact liên quan → LLM phân loại quan hệ candidate↔existing thành NEW / DUPLICATE / UPDATE / CONTRADICT.
• CONTRADICT: fact cũ valid_to=now, status='superseded'; fact mới insert status='active'.
• UPDATE: cùng predicate, object đổi (đổi tên interest) → tương tự CONTRADICT nhưng giữ liên kết.
• DUPLICATE: bỏ qua hoặc tăng confidence.
• NEW: insert thẳng.
Vấn đề giải quyếtPainpoint #2 — bài toán "dị ứng bò → hết dị ứng bò". Temporal invalidation thay vì xoá → giữ audit, rollback được.
Bẫy cần tránh quan trọngSemantic search bắt cả "dị ứng bò" ↔ "hết dị ứng bò" (vector rất giống) NHƯNG đó là mâu thuẫn, không phải trùng lặp → phải để LLM phân loại rõ, không dựa cosine. Ngược lại "thích RAG" + "thích GraphRAG" là cùng đúng, LLM hay over-trigger CONTRADICT → cần threshold confidence + prompt few-shot.
REL = classify(candidate, existing_top_k)  # LLM, temp=0
if REL == "CONTRADICT" or REL == "UPDATE":
    supersede(existing.id, valid_to=now)          # status='superseded'
    insert(candidate, status='active', valid_from=now)
elif REL == "DUPLICATE":
    bump_confidence(existing.id)                    # không insert bản mới
else:  # NEW
    insert(candidate, status='active', valid_from=now)

R2 — persona_context.build() làm mới

Công nghệ/APIThuần Python + đọc DB — KHÔNG gọi LLM (LLM chỉ chạy ở batch đêm). Không phải bảng lưu sẵn — render động tại request. MVP: build mỗi request, KHÔNG cache (chỉ là 1–2 SELECT có index, cỡ mili-giây; cache thêm bug invalidation, để dành khi đo thấy chậm).
Thuật toánĐọc user_facts(active) + user_topic_interests (interest đã có sẵn) → 3 tầng: Core (tên + top 3–5 interest + preference chính, luôn inject) · Ranking (khi vượt ngưỡng chọn theo priority(type)×confidence×recency, identity>preference>interest) · Archival (fact còn lại chỉ semantic-search kéo ra khi câu hỏi liên quan). Hard cap ~400 token.
Vấn đề giải quyếtChống phình prompt khi fact nhiều (mô hình core vs archival memory của MemGPT). Trộn 2 nguồn: fact (lời nói) + interest (hành vi).

Có 2 cách đưa fact vào prompt. Cách 2 (index nhẹ luôn nhét → tra subject nặng khi cần) chính là core/archival của MemGPT — đúng đích nhưng over-built cho v1 vì persona 1 user còn nhỏ.

5. Phương pháp test, đánh giá so sánh

  • Unit fact_extractor: câu ephemeral ("hôm nay tui mệt") → KHÔNG sinh fact; câu "tui tên Khoa" → fact identity confidence cao
  • Unit reconciler — CONTRADICT: seed "dị ứng bò" active → nạp "hết dị ứng bò" → fact cũ status='superseded'+valid_to set, fact mới active
  • Unit reconciler — không over-trigger: "thích RAG" + "thích GraphRAG" → 2 fact cùng active, KHÔNG bị đánh CONTRADICT
  • Unit reconciler — DUPLICATE: nạp lại y hệt fact đã có → không tạo bản mới, chỉ bump confidence
  • persona_context budget: seed 50 fact → output ≤ token cap, giữ đúng thứ tự priority (identity trước interest)
  • Inject an toàn: fact chứa chuỗi "ignore previous instructions" → không đổi hành vi bot (đóng khung context-only)
  • Greeting E2E: user đã có fact tên "Khoa" → lời chào có "Khoa"; user chưa có tên → chào generic, không bịa tên
  • Fail-open: persona_context.build ném lỗi → chat vẫn trả lời bình thường (persona="")
  • Delta batch: chạy job 2 lần liên tiếp không có message mới → lần 2 không gọi LLM (idempotent theo last_processed_message_at)
  • Scheduler: mock clock tới 03:00/05:00 → job chạy đúng 1 lần, không chồng lấn
  • Vector thật vs SequenceMatcher: bộ 20 cặp (topic, paper) gán nhãn tay → so precision@5 giữa 2 cách chấm
  • Notification dedup: chạy monitor 2 lần cùng ngày → cùng paper không tạo 2 notification (unique constraint)
  • Privacy: user xoá fact qua Memory UI → lần chat kế tiếp persona không còn fact đó

7. API & Schema chi tiết

7.1 DDL — bảng mới

-- Fact struct + temporal (mô hình Zep) + provenance
CREATE TABLE public.user_facts (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       uuid NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
  fact_type     text NOT NULL CHECK (fact_type IN
                  ('identity','interest','preference','constraint')),
  subject       text NOT NULL DEFAULT 'user',
  predicate     text NOT NULL,          -- is_named / researches_topic / allergic_to ...
  object        text NOT NULL,
  source        text NOT NULL DEFAULT 'extracted'   -- 'user' (form) thắng 'extracted' (LLM) khi mâu thuẫn
                  CHECK (source IN ('user','extracted')),
  tier          text NOT NULL DEFAULT 'core'        -- 'core' luôn nhét | 'detail' kéo khi cần (v2)
                  CHECK (tier IN ('core','detail')),
  embedding     jsonb,                  -- để semantic dedup/contradiction
  confidence    numeric NOT NULL DEFAULT 0.5,
  status        text NOT NULL DEFAULT 'active'
                  CHECK (status IN ('active','superseded','retracted')),
  valid_from    timestamptz NOT NULL DEFAULT NOW(),
  valid_to      timestamptz,            -- NULL = còn hiệu lực
  source_message_id uuid REFERENCES public.messages(id) ON DELETE SET NULL,
  created_at    timestamptz NOT NULL DEFAULT NOW(),
  updated_at    timestamptz NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_user_facts_active ON public.user_facts (user_id, status, fact_type);
-- RLS: SELECT own; INSERT/UPDATE chỉ service role (batch đêm), giống notifications
ALTER TABLE public.user_facts ENABLE ROW LEVEL SECURITY;
CREATE POLICY user_facts_select_own ON public.user_facts
  FOR SELECT USING (auth.uid() = user_id);

-- Con trỏ delta để batch không đọc lại toàn bộ lịch sử
CREATE TABLE public.fact_extraction_runs (
  user_id                   uuid PRIMARY KEY REFERENCES public.profiles(id) ON DELETE CASCADE,
  last_processed_message_at timestamptz,
  last_run_at               timestamptz,
  stats                     jsonb NOT NULL DEFAULT '{}'
);
